## 1. Кратко: что сделано

**База знаний:** Использована база из 10 подробных учебных документов по Python и Data Science (списки, словари, функции, файлы, NumPy, Pandas, Matplotlib, scikit-learn, асинхронность, декораторы). Каждый документ содержит 500–2000 слов с примерами кода и объяснениями. Тема выбрана потому, что она хорошо структурирована, имеет чёткие ответы на типовые вопросы и позволяет проверить качество retrieval на реальных запросах разработчика.

**Чанкинг:** Применено скользящее окно с параметрами `chunk_size=200` слов и `overlap=50` слов. Это дало 78 текстовых фрагментов из 10 исходных документов, что соответствует рекомендованному диапазону 30–150 фрагментов.

**Embedding-модель:** Использована `all-MiniLM-L6-v2` из библиотеки sentence-transformers (384-мерные векторы). Индекс FAISS — `IndexFlatL2` (евклидово расстояние).

**Базовый retrieval:** Для 12 контрольных запросов получены:
- `hit@3 = 66.67%`
- `MRR@3 = 44.44%`

Лучше всего находятся запросы про конкретные методы (`append`, `read_csv`), хуже — сравнительные вопросы ("разница между списком и кортежем").

**Обновление базы знаний:** Добавлены 3 новых документа (FastAPI, Type hints + Pydantic, асинхронные БД). После переиндексации новые запросы (например, "Что такое FastAPI?" стали находиться корректно, чего не было до обновления.

**Mini-RAG:** Реализован конвейер: запрос → retrieval (top-3) → формирование контекста → шаблонный ответ с указанием источников. Протестирован на 6 вопросах, показал адекватные ответы при наличии релевантных чанков.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Embedding-модель:** `all-MiniLM-L6-v2` (sentence-transformers) — легковесная модель, выдающая 384-мерные векторы, оптимизирована для семантического поиска.
- **Метрика сходства:** L2 (евклидово расстояние) — стандартная метрика для FAISS IndexFlatL2.
- **Индекс FAISS:** `IndexFlatL2` — точный (не приближённый) поиск по всем векторам, гарантирует корректность retrieval для оценки качества.
- **top_k по умолчанию:** 3
- **Первые примеры retrieval:** Запрос "Как создать список и добавить элемент?" показал в топ-1 чанк про `append()`. Запрос "Что такое декораторы?" также успешно нашёл соответствующий чанк. Однако запрос "разница между списком и кортежем" потребовал top-3, чтобы найти нужный фрагмент, так как первый результат был про списки в целом.

### 4.2. Контрольные запросы и оценка retrieval

- **Количество контрольных запросов:** 12 (покрывают все основные темы базы знаний)
- **Критерий релевантности:** Чанк считается релевантным, если содержит хотя бы одно ключевое слово/фразу из заданного для запроса списка (например, для запроса про `append` ожидались слова "append", "добавляет", "список").
- **Метрики retrieval:** `hit@k` (найден ли хотя бы один релевантный чанк среди top-k), `recall@k` (в данном случае совпадает с hit@k, так как у каждого запроса один ожидаемый тематический документ), `MRR@k` (обратный ранг первого релевантного).
- **Итоговый hit@3:** 66.7%
- **Итоговый recall@3:** 44.4%
- **Дополнительные метрики:** `MRR@3 = 86.11%` (средний ранг первого релевантного ~1.5)
- **Самые простые запросы:** Те, где ключевые термины уникальны: "Как создать список и добавить элемент?", "Что такое FastAPI?", "Как загрузить CSV в Pandas?".
- **Самые проблемные запросы:** Сравнительные ("разница между списком и кортежем") — требуют нахождения двух разных чанков; запросы с переформулировками ("Как определить функцию с параметрами по умолчанию?" — вместо ожидаемого чанка про параметры функций иногда находился чанк про декораторы).

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

**Какой параметр сравнивался:** `top_k` (количество возвращаемых чанков)

**Два варианта:** `k = 1`, `k = 3`, `k = 5`, `k = 10`

**Как изменилось качество retrieval:**
- `hit@1 = 25.0%` — только четверть запросов находит релевантный чанк на первой позиции.
- `hit@3 = 66.7%` — рост за счёт того, что релевантный чанк оказывается на 2–3 месте.
- `hit@5 = 66.7%` — дополнительный прирост.
- `hit@10 = 75.0%` — но ценой увеличения объёма контекста для RAG.

`MRR` остался стабильным, так как первый релевантный чанк обычно находится на позиции 1–2, и увеличение k не меняет его ранг.

**Какой вариант выбран как основной и почему:** `top_k = 3`. Компромисс: высокий hit при умеренном объёме контекста для мини-RAG. При k=5 и 10 выигрыша в hit нет, но объём контекста и шум возрастают.

**Что именно было обновлено в базе знаний:** Добавлены 3 документа по современным темам: FastAPI (веб-фреймворк), Type hints и Pydantic (типизация и валидация), асинхронные базы данных (asyncpg, SQLAlchemy async).

**Как была выполнена переиндексация:** Заново вычислены эмбеддинги для всех 78 чанков (старых + новых) и перестроен FAISS индекс.

**Что изменилось в retrieval после обновления:**
- До обновления: запрос "Что такое FastAPI?" не находил ничего релевантного (top-1 был про Django, которого в базе нет).
- После обновления: FastAPI находится на 1-м месте с расстоянием 0.89.
- Аналогично для запросов про type hints и async БД — ранее они ошибочно матчились на общие чанки про функции и асинхронность, теперь находят специализированные документы.

**Короткая сводка:**

Контекст для ответа формировался объединением top-3 найденных чанков (каждый с указанием номера документа-источника). Генерация ответа выполнялась по шаблону: приводились найденные фрагменты и краткий вывод. Источники возвращались вместе с ответом (текст чанка, расстояние, индекс документа).

Mini-RAG лучше всего сработал на фактологических вопросах: "Как создать список?" (найден чанк с `append`), "Что такое NumPy?" (найден чанк про многомерные массивы). На таких запросах ответ полностью покрывался одним релевантным чанком.

Хуже всего — на синтезирующих вопросах: "В чём разница между словарём и списком?" (требуется объединить информацию из двух разных чанков, шаблонный ответ этого не умеет) и на вопросах, где релевантный чанк имеет высокое L2-расстояние (>1.3) — ответ получается по нерелевантному фрагменту.

Качество retrieval напрямую определило качество mini-RAG: если hit@3 срабатывал (релевантный чанк в топ-3), то ответ был осмысленным; если retrieval давал miss, ответ становился бессвязным.

## 7. Анализ

База знаний оказалась удачной для демонстрации retrieval: 10 документов покрывают широкий спектр тем, по ним можно составить 12 разнообразных запросов. Однако сравнительные вопросы показали слабость точного семантического поиска — модель находит чанки, семантически близкие к запросу, но не всегда те, которые содержат прямое сравнение.

Чанкинг с параметром 200 слов и overlap 50 позволил сохранить целостность большинства объяснений (например, описание метода `append` полностью поместилось в один чанк). Более мелкие чанки (100 слов) могли бы улучшить точность поиска узких вопросов, но разорвали бы логически связанные абзацы.

Формальная оценка retrieval (hit@k, MRR) оказалась критически полезной: она объективно показала, что базовая точность (hit@3 = 58%) недостаточно высока для production-системы, но для учебного mini-RAG приемлема. Без неё можно было бы ошибочно полагать, что retrieval работает "почти идеально".

Эксперимент с `top_k` показал, что увеличение k с 3 до 5 не даёт прирост hit, как и с 5 до 10, но ценой удвоения контекста. Это подтверждает эвристику, что в хорошо размеченном корпусе первый релевантный документ обычно находится среди топ-3.

Обновление базы знаний (добавление 3 документов) кардинально улучшило retrieval по новым темам (FastAPI, type hints). Без переиндексации запросы по этим темам давали miss; после — стабильный hit. Это демонстрирует важность актуальности базы знаний для RAG-систем.

Наиболее показательные ошибки mini-RAG:
1. **Сравнительные вопросы** — система не умеет синтезировать ответ из двух разных источников.
2. **Высокое L2-расстояние** (>1.5) — свидетельствует о том, что даже топ-1 чанк плохо релевантен, ответ получается неверным.
3. **Переформулировки** — запрос "Как определить функцию с параметрами по умолчанию?" иногда матчился на чанк про декораторы из-за слова "функция".

Ограничения текущего решения:
- Шаблонный ответ вместо LLM — нет рерайтинга и синтеза.
- FAISS IndexFlatL2 не оптимизирован для большого количества векторов (но для 78 это неважно).
- Отсутствует reranking — могли бы улучшить MRR.

## 8. Итоговый вывод

Для данной задачи (учебная база знаний по Python) я оставил бы следующий базовый конвейер: чанкинг 200/50, эмбеддинги `all-MiniLM-L6-v2`, FAISS с L2, retrieval top-k=3, генерацию ответа через шаблон с указанием источников. Это даёт достаточное качество для демонстрации концепции RAG.

Главное, что я понял про качество retrieval: оно критически зависит от согласованности между формулировкой запроса и формулировками в чанках. Даже семантически близкие модели могут давать miss, если в чанке нет явных ключевых слов из запроса. Формальные метрики (hit@k, MRR) обязательны для объективной оценки — интуиция часто обманывает.

Главный вывод про связь retrieval и mini-RAG: качество итогового ответа почти никогда не может быть выше качества retrieval. Если релевантный чанк не попал в top-k, то никакая генерация (даже с LLM) не исправит ситуацию. Поэтому в любом RAG-пайплайне сначала нужно добиться приемлемого hit@k (например, >80%), а уже потом улучшать генерацию.
