## 2. Постановка задачи и контекст

### 2.1. Предметная область и задача

**Задача:** бинарная классификация – предсказать, прекратит ли клиент пользоваться услугами компании в следующем месяце.

**Пользователи сервиса:** менеджеры по работе с клиентами, маркетологи, аналитики. Сценарий использования: сотрудник вводит данные клиента (или система автоматически подгружает их из CRM), получает оценку риска и принимает решение о необходимости удержания (скидка, звонок, специальное предложение).

### 2.2. Формулировка в терминах ML

- **Входные данные:** демографические признаки (пол, возрастная группа), информация об услугах (телефон, интернет, доп. сервисы), данные о контракте, платежах (ежемесячный счёт, общая сумма), длительность обслуживания.
- **Целевая переменная:** `Churn` – бинарный флаг (1 – клиент ушёл, 0 – остался).
- **Ограничения:** интерпретируемость важна для бизнеса (чтобы понять причины оттока), приемлемое время ответа (< 100 мс), модель должна быть устойчива к несбалансированным данным (отток составляет ~26%).

### 2.3. Целевые метрики

- **AUC-ROC** – основная метрика, так как классы несбалансированы, а нам важна способность модели различать классы при любом пороге.
- **F1-мера** – компромисс между полнотой и точностью, важна для бизнеса (чтобы не упустить уходящих клиентов и не тратить ресурсы на ложные срабатывания).
- **Precision / Recall** – дополнительно для анализа работы модели.

## 3. Данные

### 3.1. Источник данных

Открытый датасет **Telco Customer Churn** с Kaggle:  
[https://www.kaggle.com/datasets/blastchar/telco-customer-churn](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)

### 3.2. Структура данных

- 7043 строки, 21 столбец.
- **Категориальные признаки:** `gender`, `Partner`, `Dependents`, `PhoneService`, `MultipleLines`, `InternetService`, `OnlineSecurity`, `OnlineBackup`, `DeviceProtection`, `TechSupport`, `StreamingTV`, `StreamingMovies`, `Contract`, `PaperlessBilling`, `PaymentMethod`.
- **Числовые признаки:** `SeniorCitizen`, `tenure`, `MonthlyCharges`, `TotalCharges`.
- **Целевая переменная:** `Churn` (Yes/No).

### 3.3. Предобработка и EDA

**Шаги предобработки (notebook `exp01_data_eda_preprocessing.ipynb`):**
- Приведение `TotalCharges` к числовому типу, заполнение пропусков нулями.
- Удаление `customerID` (неинформативный).
- Создание новых признаков:
  - `tenure_group` – группировка срока обслуживания.
  - `TotalServices` – количество подключённых услуг.
  - `Charges_Tenure_Interaction` – произведение `MonthlyCharges` и `tenure`.

**Ключевые выводы EDA:**
- Дисбаланс классов: 26.5% оттока.
- Помесячный контракт → отток ~45%.
- Низкий tenure (<12 мес) → высокий отток.
- Высокие `MonthlyCharges` коррелируют с оттоком.
- Электронные чеки, отсутствие `OnlineSecurity`/`TechSupport` увеличивают риск.

## 4. Модели и подходы

### 4.1. Базовые модели (exp02_baseline_models.ipynb)

- **Logistic Regression** (max_iter=1000)
- **Random Forest** (100 деревьев)
- **XGBoost** (100 деревьев, lr=0.1)

**Результаты на тестовой выборке:**

| Модель                 | AUC-ROC | F1-score |
|------------------------|---------|----------|
| Logistic Regression    | 0.8428  | 0.587    |
| Random Forest          | 0.8210  | 0.562    |
| XGBoost                | 0.8425  | 0.581    |

Логистическая регрессия показала лучший AUC-ROC и была выбрана как baseline.

### 4.2. Нейросетевой базлайн (exp03_neural_baseline.ipynb)

- **MLPClassifier** с одним скрытым слоем (32 нейрона) – выбрано через кросс-валидацию.
- Результат: AUC-ROC = 0.8226, что ниже, чем у логистической регрессии. Нейросеть не превзошла простую линейную модель.

### 4.3. Улучшение моделей (exp04_model_tuning.ipynb)

Проведён **RandomizedSearchCV** для XGBoost по сетке:
- `n_estimators`: [100, 200, 300]
- `max_depth`: [3, 6, 9, 12]
- `learning_rate`: [0.01, 0.05, 0.1, 0.2]
- `subsample`, `colsample_bytree`, `reg_alpha`, `reg_lambda`.

**Лучшие параметры:**
```python
{'subsample': 0.7, 'reg_lambda': 1.0, 'reg_alpha': 0.1,
 'n_estimators': 100, 'max_depth': 3, 'learning_rate': 0.05,
 'colsample_bytree': 0.8}
```

Улучшенный XGBoost показал AUC-ROC = 0.8480 (незначительное улучшение по сравнению с базовой логистической регрессией).

## 5. Экспериментальный протокол и результаты

### 5.1. Протокол

- Данные разделены на train (80%) и test (20%) со стратификацией по `Churn`.
- Для всех моделей использовалась одна и та же предобработка (one-hot кодирование, стандартизация).
- Метрики считались на отложенной тестовой выборке.
- Для XGBoost дополнительно применялась 3‑fold кросс-валидация при поиске гиперпараметров.

### 5.2. Сравнение моделей

| Модель                          | AUC-ROC | F1     | Precision | Recall |
|--------------------------------|---------|--------|-----------|--------|
| Logistic Regression (baseline) | 0.8428  | 0.5872 | 0.6633    | 0.5267 |
| Random Forest                  | 0.8210  | 0.5625 | 0.6342    | 0.5053 |
| XGBoost (базовый)              | 0.8425  | 0.5808 | 0.6246    | 0.5428 |
| XGBoost (улучшенный)           | 0.8480  | 0.5658 | 0.653     | 0.5348 |
| MLP (32 нейрона)              | 0.8226  | 0.5727 | 0.623     | 0.529  |

### 5.3. Выбор финальной модели

Финальной моделью выбрана **логистическая регрессия**, так как:
- Она показала AUC-ROC = 0.8428, что лишь незначительно уступает улучшенному XGBoost (0.8480), но при этом:
  - Проще интерпретируется (коэффициенты указывают на влияние каждого признака).
  - Обучение и инференс выполняются быстрее.
  - Менее подвержена переобучению на малом объёме данных.
- Компромисс: небольшое снижение AUC-ROC оправдано простотой и стабильностью в продакшене.

## 6. Архитектура решения и сервис

### 6.1. Архитектура пайплайна

1. **Исходные данные** → загрузка CSV.
2. **Предобработка** (очистка, создание признаков) – скрипты в ноутбуках, сохранённый `preprocessor.joblib`.
3. **Обучение** – логистическая регрессия, сохранение модели в `artifacts/`.
4. **Сервис (FastAPI)** – загружает препроцессор и модель, принимает JSON, возвращает предсказание.

### 6.2. API и endpoints

- `GET /health` – проверка готовности сервиса.
- `POST /predict` – принимает JSON с признаками клиента (полный набор, аналогичный исходным данным без `customerID` и `Churn`).  
  **Ответ:** `{"churn_probability": float, "risk_category": "low"|"medium"|"high"}`.
- `GET /metrics` – служебные счётчики (количество запросов, ошибок).

**Документация:** автоматическая Swagger UI по адресу `/docs`.

### 6.3. Технологический стек

- **Фреймворки:** FastAPI, Uvicorn.
- **ML библиотеки:** scikit-learn, XGBoost, pandas, numpy, joblib.
- **Эксперименты:** MLflow (логирование параметров и метрик).
- **Контейнеризация:** (планируется, но пока не реализована) – в проекте нет Dockerfile, только локальный запуск.
- **Окружение:** Python 3.10+.

## 8. Ограничения и дальнейшая работа

**Текущие ограничения:**
- Модель не учитывает временные тренды (сезонность, акции конкурентов).
- API не авторизован – может использоваться только во внутреннем контуре.
- Отсутствует контейнеризация, что усложняет развёртывание.

**Что можно улучшить:**
- **Модель:** применить CatBoost, LightGBM, попробовать стекинг. Улучшить балансировку классов (SMOTE, взвешивание).
- **Сервис:** добавить кэширование для повторяющихся запросов, реализовать асинхронную обработку, написать Dockerfile и docker-compose.
- **Данные:** собрать больше признаков (поведенческих, из call‑центра), добавить историю оттока по когортам.
- **Наблюдаемость:** интегрировать с Prometheus/Grafana, добавить алерты.
